Production Page Object Rules

Production UI tests use pages/, pages/components/, and flows/.

  • 🏗️ Page objects initialize stable locator members in __init__.
  • ♻️ Methods reuse those members instead of rebuilding the same locator repeatedly.
  • 🎯 Dynamic locators that depend on runtime input, such as a product name, stay inside the method that receives that input.
  • 🛡️ CSS-heavy locators use deterministic self-healing fallbacks through pages/self_healing.py.
  • 📋 Self-healing does not call AI or invent selectors. Fallbacks are explicitly owned by the page object and recorded through self_heal_events() so stale primary locators can be fixed.

Install all dependencies in one go:

pip install -r requirements.txt
playwright install chromium

To run registration tests, create data/secrets.json from data/secrets.json.example and fill in real credentials.

Secrets & Test Data

Secrets (data/secrets.json)

Credentials are runtime-only — never commit data/secrets.json.

WhereHow secrets are supplied
LocalCopy data/secrets.json.example → data/secrets.json and fill in real values. Listed in .gitignore.
CIGitHub Actions secret TEST_SECRETS_JSON (full JSON body). Workflow writes it to data/secrets.json on the runner only; the step does not print the secret.
DockerMount data/secrets.json read-only at run time (scripts/run_all_tests_docker.sh); excluded via .dockerignore.

utils/data_loader.py merges secrets.json over data/test_data.json when the file exists. Registration tests skip via registration_data if secrets are missing — they do not fail the whole suite.

Data-driven parametrize corpora

Data-driven tests load their parametrize corpus via get_data_file(filename), which reads any JSON file from data/ by name.

FileUsed by
data/api_test_cases.json tests/api/test_api_data_driven.py — search, price-filter, cart, and empty-search cases
data/pentest_cases.json tests/api/test_pentest_data_driven.py — injection, auth attacks, access-control routes, and file-exposure vectors
⚠️

Before pushing: git check-ignore -v data/secrets.json and git status must not list data/secrets.json. Capstone deliverables must not include real passwords or API keys.

Running Tests

⚡

pytest.ini runs tests in parallel by default: -n auto --dist loadgroup. All web-UI tests share a single worker (via xdist_group("web-ui") in tests/web-ui/conftest.py) to avoid concurrent headless-browser bot-protection triggers. API, contract, and unit tests are distributed across all workers. CI uses pytest tests/ -m "not demo" to skip teaching-only tests.

🌐

Offline fallback: All HTTP requests to tutorialsninja.com are transparently intercepted by services/api/opencart_fallback.py via services/rest_client.py. The fallback simulates a complete OpenCart store — search results, cart state, home page, login, and registration — without a real network connection. Web-UI tests (Playwright) still use a real browser and need the live site.

Docker & Compose

Build & run

Build image
docker build -t ai-automation-testing .
Full suite
docker run --rm \
  -v "$PWD/docker-artifacts:/app/test-artifacts" \
  ai-automation-testing pytest tests/ \
  --alluredir=/app/test-artifacts/allure-results
Helper script (builds + mounts secrets)
./scripts/run_all_tests_docker.sh

Compose stack

Starts database, automation server, Prometheus, Grafana, runs all tests, and generates a fresh Allure report.

Via script
./scripts/run_compose_tests_with_allure.sh
Via npm
npm run test:compose:report
Pass extra args
./scripts/run_compose_tests_with_allure.sh \
  tests/api tests/contract -q

Compose service URLs

Grafana http://localhost:3000/d/automation/automation-runs
Prometheus http://localhost:9090
API docs http://localhost:8000/docs
Allure opened automatically from docker-artifacts/allure-report

With secrets (registration tests)

docker run --rm \
  -v "$PWD/data/secrets.json:/app/data/secrets.json:ro" \
  -v "$PWD/docker-artifacts:/app/test-artifacts" \
  ai-automation-testing pytest tests/ \
  --alluredir=/app/test-artifacts/allure-results

Allure Reports

Results are written to allure-results/. This project uses Allure Report 3 through the local npm package — avoid the global allure command if it reports a 2.x version.

Check version
npm run allure:version
Generate report
npm run allure:generate
Open in browser
npm run allure:open
🔄

For live report updates while rerunning tests: npm run allure:watch
