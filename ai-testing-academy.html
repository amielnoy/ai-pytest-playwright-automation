01אוטומציית בדיקות UI — Playwright + POM

הבסיס: בדיקות דפדפן על אתר חנות דמו (tutorialsninja.com/demo), כשכל אינטראקציה עם הדף עוברת דרך Page Objects.

העיקרון: סלקטורים לא מופיעים בבדיקות

כל דף באתר מיוצג במחלקה תחת pages/ שיורשת מ-BasePage. הבדיקה קוראת למתודות עסקיות בלבד — כשהאתר משתנה, מתקנים שורה אחת ב-Page Object במקום עשרים בדיקות.

# tests/web-ui/test_registration.py — כך נראית בדיקה אמיתית
@allure.feature("Registration")
@pytest.mark.registration
class TestRegistration:

    @allure.title("Successful new-user registration")
    def test_register_new_user(self, registration_pages, registration_data):
        registration_pages.home.open()
        with allure.step("Navigate to Register page"):
            registration_pages.home.go_to_register()
        with allure.step("Fill registration form"):
            registration_pages.register.register(**registration_data)

Fixtures — הזרקת התלויות של pytest

קובץ conftest.py בשורש מגדיר את שרשרת הדפדפן: browser_instance (אחד לכל הריצה) ← context (חדש לכל בדיקה, לבידוד) ← page. Hook של pytest_runtest_makereport מצרף צילום מסך אוטומטי לכל כישלון.

נתונים וקונפיגורציה

  • config/config.json — הגדרות דפדפן (headless, timeout, viewport).
  • data/test_data.json — כל קלטי הבדיקות, עם placeholder בשם {ts} שהופך ל-timestamp כדי שכל ריצה תקבל אימייל ייחודי.
  • utils/data_loader.py — נקודת הכניסה היחידה לקריאת קונפיגורציה ונתונים.

עמידות: Self-Healing Locators

pages/self_healing.py מממש ריפוי-עצמי דטרמיניסטי: ל-locator ראשי מוגדרת רשימת fallbacks מאושרת מראש, וכל "ריפוי" נרשם כ-SelfHealEvent לניקוי סלקטורים בהמשך. בלי ניחושים של AI — הכול בשליטת ה-Page Object.

02פירמידת הבדיקות — ארבע שכבות

בדיקות UI הן היקרות והאיטיות ביותר. רוב הכיסוי צריך לשבת בשכבות מהירות יותר.

⚡ Unit — tests/unit

בלי Playwright, בלי רשת, בלי סודות. Mock על גבולות ה-session. דוגמאות: test_rest_client.py, test_self_healing.py.

🔌 API — tests/api

דרך שכבת services בלבד: rest_client.py (retries, timeout) + מחלקות שירות כמו search_service.py. אין requests ישיר בבדיקות ואין קודי סטטוס קשיחים.

📜 Contract — tests/contract

מוודאות חוזים יציבים: קודי סטטוס, שדות חובה, מזהי מוצר ומחירים ניתנים לפירסור. מונחות-מפה — צורת assertion אחת להרבה endpoints.

🖥️ Web-UI — tests/web-ui

מעט בדיקות, על התרחישים הקריטיים. Setup איטי עובר ל-API (למשל מילוי עגלה) במקום קליקים בדפדפן.

הרצה לפי שכבה או marker

pytest tests/unit                 # מהיר — רץ בכל מקום
pytest -m registration            # לפי marker מ-pytest.ini
pytest -m cart
pytest tests/web-ui/test_registration.py::TestRegistration::test_register_new_user

03ה-DevOps הנדרש

בדיקות שרצות רק על הלפטופ שלך לא שוות הרבה. כך הפרויקט רץ בכל מקום, באופן זהה.

Docker — סביבה זהה בכל מכונה

docker build -t ai-automation-testing .
docker run --rm -v "$PWD/docker-artifacts:/app/test-artifacts" \
  ai-automation-testing pytest tests/ --alluredir=/app/test-artifacts/allure-results

Docker Compose — סטאק שלם בפקודה אחת

docker-compose.automation.yml מרים את הבדיקות יחד עם Prometheus, Grafana, Swagger UI ו-cAdvisor:

npm run test:compose:report

עקרונות שנאכפים בריפו

  • גרסאות נעוצות ב-requirements.txt — ריצות CI שחזוריות.
  • תוצרים מחוץ ל-git: allure-results/, logs/, caches וסודות (data/secrets.json עם קובץ example בלבד).
  • סנכרון תיעוד: כל שינוי ב-Docker מחייב עדכון README, סקריפטים ו-GitHub Actions יחד.
  • שירות חדש ב-Compose? מעדכנים גם את monitoring/prometheus.yml (scrape targets) ואת שני סקריפטי ה-compose.

04שרת תוצאות בדיקות — Allure + ניטור

תוצאה של ריצת בדיקות היא מוצר בפני עצמו: דוח קריא, מדדים לאורך זמן, והתראות.

Allure — הדוח לכל ריצה

  • כל בדיקה מקושטת ב-@allure.feature / @allure.story / @allure.title / @allure.severity.
  • צעדים לוגיים עם with allure.step(...).
  • לוגים מצורפים אוטומטית דרך fixture בשם log (מ-utils/logger.py), וצילום מסך על כל כישלון.
npm run allure:generate   # בונה דוח מ-allure-results
npm run allure:open       # פותח בדפדפן

ניטור מתמשך — Grafana + Prometheus

שירותכתובתתפקיד
Grafana — Test Runslocalhost:3000/d/automation/automation-runsמגמות הצלחה/כישלון של ריצות
Grafana — Infrastructurelocalhost:3000/d/infra-networkקצב בקשות, latency, שגיאות HTTP
Prometheuslocalhost:9090איסוף מדדים
Swagger UIlocalhost:8090תיעוד ה-API מ-docs/openapi.yaml

מדדי HTTP נאספים אוטומטית ב-server/app.py באמצעות prometheus-fastapi-instrumentator.

05בדיקות בעזרת AI — לשני הכיוונים

כיוון א': המוצר שלך מכיל LLM — איך בודקים אותו דטרמיניסטית. כיוון ב': ה-AI עוזר לך לבדוק.

א. בודקים את ה-AI

  • Prompt Injection — tests/api/test_prompt_injection.py משתמש בטכניקת SENTINEL: מגדירים מחרוזת שתופיע בתשובה רק אם המתקפה הצליחה, ובודקים שהיא נעדרת. כישלון בבדיקה = הוכחה להזרקה אמיתית.
  • DLP — tests/api/test_chat_dlp.py מוודא שסודות ו-PII לא דולפים בתשובות.
  • פונקציונליות — tests/api/test_chat_ai.py מול שרת FastAPI מקומי; ללא ANTHROPIC_API_KEY השרת מחזיר תשובות mock דטרמיניסטיות, כך שהבדיקות רצות בכל מקום.

ב. ה-AI בודק בשבילנו

  • סוכן ניתוח כשלים — agents/allure_failure_agent.py קורא את allure-results/ ומסווג כל כישלון: product_bug / dom_change / automation_bug / environment_issue / flaky. הרצה: ./scripts/run_allure_failure_agent.sh (או --no-ai למצב היוריסטי).
  • סוכן מתכנן בדיקות — agents/test_planner_agent.py: Playwright סורק דף חי ל-JSON מובנה, ו-Claude כותב מסמך STD תוך קריאת הבדיקות הקיימות כדי לא לשכפל כיסוי.
  • פקודות Slash ב-Claude Code — /generate-test, /review-tests, /create-page-object (דרך Playwright MCP), /fix-selectors, /analyze-failures.
  • העיקרון: הסוכן מציע — המהנדס מאשר. וסוכנים הם תוכנה: יש להם בדיקות יחידה משלהם (tests/unit/test_allure_failure_agent.py).
