Automation Quality Gates

Automation work is not ready for review until it satisfies these gates:

  • Tests describe business behavior rather than Playwright or HTTP mechanics.
  • New UI tests use page objects from pages/.
  • Raw selectors do not appear in new test files.
  • Locators follow Playwright priority: role, label, placeholder/text, test id, then attribute fallback.
  • Stable page object and component locators are initialized in constructors as instance members.
  • Dynamic locators stay inside methods only when they depend on runtime input.
  • CSS-heavy locators use deterministic self-healing fallbacks only when the fallback is explicit and reviewable.
  • Setup is explicit through fixtures or API helpers.
  • Fixture scope is intentional and parallel-safe.
  • Test data is isolated, repeatable, and comes from JSON, factories, fixtures, API setup, or secrets as appropriate.
  • Assertions are specific and behavior-focused.
  • Failures produce useful evidence: message, screenshot, trace, response body, log, or Allure attachment.
  • Allure names, features, stories, and steps help a non-author understand the failure.

Before / After Examples

Locator

Weak 
page.locator("#content .btn-primary").click()
Strong 
page.get_by_role("button", name="Add to Cart").click()

Why: the strong locator follows the user-facing role/name and survives CSS refactors.

Page Object

Weak 
def test_add_to_cart(page):
    page.get_by_placeholder("Search").fill("iPod")
    page.get_by_role("button", name="Search").click()
Strong 
def test_add_to_cart(home_page, search_results_page):
    home_page.search("iPod")
    search_results_page.add_product_to_cart("iPod Classic")

Why: the test says what behavior is being verified; page mechanics stay in page objects.

Fixture

Weak 
def test_checkout(page):
    login(page)
    add_item(page)
Strong 
def test_checkout(logged_in_page, cart_with_item):
    cart_with_item.proceed_to_checkout()

Why: setup is explicit and reusable, and the test starts from a named state.

Review Comment

Weak 
Fix this flaky test.
Strong 
The test uses `nth(1)` to choose a product, so it can add the wrong item when sorting changes. Filter the product card by product name inside the page object.

Why: a good review finding names the risk, impact, and concrete fix.

Common Student Mistakes

  • Committing raw codegen output.
  • Using CSS selectors before role or label locators.
  • Hiding important setup in autouse fixtures.
  • Sharing mutable data across parallel tests.
  • Adding sleeps instead of waiting for state.
  • Rebuilding the same locator repeatedly inside page object methods instead of using constructor members.
  • Using self-healing to hide weak locator strategy instead of fixing stale primary selectors.
  • Accepting AI-generated tests without verifying product behavior.
  • Editing course/framework/ for capstone deliverables instead of production directories.
  • Mixing unrelated changes in one branch or commit.

Discussion Prompts

  • Which setup belongs in a fixture, and which belongs in the test?
  • What makes a locator stable?
  • What evidence proves a failure is a product bug?
  • Which tests should not be automated?
  • What work should an AI agent never perform without approval?
  • What belongs in a page object, API client, fixture, flow, or test?

Project Checklist

Use this checklist across the course and finish it before the capstone.

  • At least one manual test case has been converted into automation work.
  • UI tests use page objects from pages/.
  • No raw selectors remain in new test files.
  • Locators follow Playwright best practices.
  • Fixtures are explicit, scoped correctly, and parallel-safe.
  • Test data comes from JSON, factories, fixtures, or secrets as appropriate.
  • At least one API setup or API assertion is used.
  • At least one failure was debugged with artifact evidence.
  • Allure labels or steps improve report readability.
  • CI commands are documented.
  • At least one AI-generated output was reviewed and corrected.
  • At least one code review finding was written with risk and fix.
  • Git workflow is clean: focused branch, reviewed diff, intentional commit.
  • Capstone changes are in production directories, not only course/.

Capstone Readiness

A student is ready for the capstone when they can:

  • Explain which test layer should cover a behavior and why.
  • Write stable Playwright tests through page objects rather than raw selectors in tests.
  • Initialize stable locator members in constructors and explain when a dynamic locator belongs in a method.
  • Use deterministic self-healing fallbacks and review any recorded heal events.
  • Build explicit pytest fixtures with safe scopes and cleanup.
  • Use API setup to reduce slow or brittle UI setup.
  • Keep test data isolated and repeatable.
  • Diagnose common flaky-test causes with evidence.
  • Produce Allure reports that are useful to non-engineers.
  • Run the suite locally, in Docker, and in CI.
  • Use AI agents for bounded testing workflows while keeping deterministic tests as release gates.
  • Review automation code with a concrete risk, impact, and fix.
